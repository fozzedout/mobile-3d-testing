# mobile-3d-testing

A Vite + Three.js playground for trying out 3D ideas in the browser, with a
focus on how they actually behave on phones: touch input, device orientation,
GPU limits, and battery-friendly rendering.

## What's inside

- **Basics** — a lit, tweakable primitive (materials, color, wireframe) as a sanity check.
- **Particle Stress Test** — a point cloud with an adjustable count, for finding where a device's GPU starts to struggle.
- **Touch & Gyro** — visualizes raw multi-touch points and `deviceorientation` sensor data (with the iOS motion-permission prompt wired up).
- **Load Model (GLTF)** — drag & drop a `.glb`/`.gltf` file to inspect it on-device (triangle/node counts, wireframe toggle).
- **Space Sim (Touch Nav)** — a free-flight 6DOF lab: a cube and sphere placed far apart, found via an on-screen bearing compass. Built to A/B touch-navigation schemes: dual floating joysticks (rate-control translate + look, with a second finger on the look stick switching to a twist-to-roll gesture) vs. gyro-look + touch-thrust (device orientation for looking, one thumb for movement). Joysticks apply a logarithmic pseudo-haptic "wall" past their soft limit and synthesize click/thud audio as a Taptic Engine stand-in. Also has four independently toggleable fixes for rate-control overshoot: a gyro fine-trim layer (relative tilt nudges look on top of the stick, instead of replacing it), a time-based rate/position hybrid on the look stick (quick flicks are precise no-momentum nudges; holding promotes to continuous turning), a rotational-inertia mode (momentum while held, fast brake on release) to compare against the default instant-stop response, and a closing-rate readout on the compass so you can see you're approaching a target too fast before you sail past it.
- **Course: Ring Race** — timed run through a fixed slalom of rings, next target shown on an Elite-style 3D scanner (a tilted-disc HUD: bearing + range from the dot's polar position, altitude from a vertical stalk to its ground-plane shadow) rather than a flat compass. Best time persists locally.
- **Course: Structure Run** — timed flight down a winding tunnel (ribbed walls + a few jutting obstacles to dodge); hitting a wall/obstacle adds a time penalty with a brief invulnerability window and a screen flash, rather than resetting your run.
- **Course: Asteroid Shower** — timed run through a continuously-respawning field of drifting rocks to a large glowing finish gate (fly through it, same crossing check as the rings); same hit-penalty-and-flash feedback as Structure Run.
- **Course: Docking Bay** — timed docking at a giant voxel space station that spins for gravity. Thread the rotating 3×1-cell mail slot (a cyan glow frame marks the mouth; green/red strips show its orientation from range), land on the green pad inside the bay — spin-gravity (a real ω²r pseudo-force away from the drum's axis; Coriolis deliberately omitted) presses you onto it. A dedicated landing instrument swaps in for the scanner once you're in the bay: a top-down pad view drawn in the pad's own rotating frame (so its drift arrow shrinking to zero literally means you're rotation-matched), a height tape with descent rate, and a HOLD progress arc — plus an emissive touchdown-shadow ring on the pad tracking directly beneath you. Because real-device testers couldn't tell whether the pad was above or below them, a screen-space pad pointer sits on top of these: a small green diamond marking the pad when it's on screen, or a bold chevron pinned to the viewport edge pointing the shortest way toward it (above/below/behind) when it isn't. And rather than a knife-edge manual touchdown, a **magnetic capture field** — a faint green column standing over the pad, pulsing when it engages — catches a slow, hands-off ship inside its reach, draws it to the seat point with a magnitude-clamped spring-damper, and rights its orientation to the pad's up axis; commanding thrust past the deadband instantly breaks the field (the same gate that releases the clamps), so the magnet and the pilot never fight, and the field has to — and comfortably does — overpower spin-gravity while it holds you. Once it has drawn you close and slow enough it centres you on the pad and the clamps engage and your reference frame *becomes the station's*: the ship is seated in station-local coordinates and carried by the drum, the view co-rotates (the bay goes visually still while the stars wheel past the slot), and the hold runs while you sit. Throttling up past a deadband releases the clamps and hands you back to the inertial frame carrying the pad's tangential velocity — switching cleanly between the rotating frame and your own, both on the way down and the way out, is the real skill being tested. Then exit the slot clean. Scraping the station costs a 2s penalty with the usual flash/knockback; the whole enter-land-depart-exit loop is timed with a local best. The station is one voxel occupancy set that serves as both the rendered shell and the collider (checked in station-local space, so the rotating walls genuinely sweep toward a hovering ship), with buried cells skipped at render time for fill-rate.
- **Course: Rotor Gates** — five spinning slotted panels (increasing spin speed) you must roll-align with before flying through; a live gauge overlays the slot's and your ship's rectangles so you can see the match forming as you approach, and each entry is rated (PERFECT/GOOD/CLIPPED, or SMASHED with a time penalty if your rotated profile wouldn't actually fit) based on the angle between them, folded for the rectangle's 180° symmetry. Built specifically to exercise the roll axis.
- **Course: Asteroids Arena** — not a timed run: a survival/score arena. Fly and fire on a spinning field of asteroids that split large → medium → small when shot (each size worth progressively more points), across waves of increasing asteroid count. Three lives, a brief invulnerability + knockback on impact, game over on the last life, best score persists locally. Every nearby rock — across all three sizes — is plotted on the same Elite-style scanner the courses use, so you can find and line up shots without the target ever being on-screen.
- **Course: Saucer Duel** — a one-on-one dogfight instead of a field of targets: a single flying saucer holds a rough standoff distance while strafing, dodges perpendicular to any of your lasers whose predicted closest approach comes too close, and fires back at you with a laser aimed with a random inaccuracy cone rather than a precise shot. Three hits destroys it (score + a kill bonus), and a slightly faster, quicker-shooting one spawns shortly after. Same three-lives/knockback/game-over/best-score structure as Asteroids Arena, and the saucer shows up on the scanner the same way the asteroids do. Its movement is holonomic — it steers by picking a world-space direction and smoothing its velocity toward it, with no orientation constraint at all, so it can reverse or dart sideways almost as fast as that smoothing allows regardless of which way it's "facing" (the mesh just turns to face you cosmetically). That's deliberate: it's the "tricky, physically-implausible" one, kept as a contrast to —
- **Course: Wedge Duel** — the same one-on-one dogfight and dodge-the-incoming-laser logic, but a conventional fighter instead of a holonomic saucer: it jousts — attack runs toward you with nose guns locked, then when it gets too close it flies past / extends away and only turns back in once it has range again for another pass (rather than trying a 180° reverse while sitting on top of you). It turns its nose toward wherever it wants to go at a capped rate (`rotateTowards`, same idea as the player's own look-rate limit) and always thrusts along its *current* heading — reversing course costs an actual turn, exactly like the player. Dodges are sized to the turn actually required plus a turn-rate boost, so a jink reads as a distinct sharp maneuver. Guns are nose-mounted and fire along the nose — the alignment gate *is* its accuracy, unlike the saucer's omnidirectional turret. Each kill makes the next one faster *and* tighter-turning. The wedge shape (a flattened triangular hull with a bright centerline stripe) makes its facing readable at a glance, and it banks visually into turns — purely cosmetic, but makes the direction change legible instead of just a slowly reorienting nose.

In both duels, ramming is not free: hull contact knocks a life off you (same knockback + invulnerability window as being shot) and deals the enemy a hit's worth of damage at the same time. All three arenas use swept segment-sphere laser hit tests (`core/swept-hit.ts`) rather than per-frame point samples — at 220 u/s a bolt covers more ground per frame than the smaller hit radii on a slow mobile frame, and a point sample would tunnel straight through.

Both arenas fire via the same `FireZone`: a triangular wedge fitted into the bottom-left corner rather than a small button floating mid-edge — findable by feel (slide a thumb until it hits both physical edges) while tapering away from the move stick's area instead of reserving a full square block. Holding it down fires continuously for as long as it's held; a swipe up additionally locks in auto-fire so the thumb is free to go back to steering, and a swipe down releases it again.

The seven courses share a `FlightRig` (the exact touch-nav controls tuned in Space Sim — same GUI, same toggles). The four timed courses use a `CourseTimer` (countdown → racing → finished, hit penalties, localStorage best time); the three arenas run their own countdown → playing → game-over state machine instead, since they're scored survival rather than a race to a finish line.

FlightRig's tuning (control scheme, sensitivity, every toggle in its GUI section) persists to localStorage and carries over between scenes and courses — dial it in once on whichever course you're testing and it's already set the same way the next time you switch, rather than resetting to defaults per scene.

- **Ship Designer (Modular)** — a different kind of testbed: a touch-first drag-and-drop ship editor rather than a flight scene. Ships are built from functional blocks — reactors, engines, fuel tanks, weapons, shields, cargo bays, crew cabins — plus light **structure** fairings that exist only to shape the exterior — attached face-to-face on an *unbounded* lattice: there is no construction grid boundary, the structure is whatever you weld together. Drag a block from the bottom icon palette (or grab one already on the ship) and the block itself follows your finger the whole way — floating with a neutral blue "carrying" tint in free space, then snapping to whichever face you're pointing at, green when it can attach there, red when it collides; release to commit, an invalid release snaps back, and dragging a block down onto the palette bar (which turns into a labelled trash target while you drag) removes it. Tapping a block selects it and raises an action bar with two-axis rotation arrows (⟳ Y yaws a footprint, ⤴ X pitches it upright) plus Duplicate and Remove; a one-finger drag on empty space still orbits. Every module type comes in four sizes (S = 1 cell, M = 2, L = 4, XL = 8), where output scales slightly better than linearly with size but mass, power draw and heat scale exactly — big modules are efficient per tonne yet demand contiguous space and a power plant to match. Structure fairings are near-zero mass with no power/thrust/cargo, so you can pad tapers and fill gaps without buying another reactor. Every block wears a pictographic icon baked onto its faces (lightning = reactor, nozzle-and-flames = engine, crosshair = weapon, crate = cargo, braced frame = structure, and so on, with a size tag in the corner) so a layout reads at a glance instead of by memorized colors, and the palette buttons use the same icons. Layouts must satisfy structure-relative rules — engines need clear space behind their exhaust, weapons can't be fully buried, at least one reactor/engine/crew cabin, a non-negative power balance, and one connected structure. Rule-breaking blocks tint red with the issues listed, and a one-tap **Repair layout** relocates stranded engines/weapons and prunes disconnected islands (it won't invent reactors for a power deficit — that trade-off is yours). An outer hull is generated around whatever's installed and visibly re-fits as the layout changes; two switchable styles: **plated** (default) merges exact occupied cells into inflated rectangular hull sections so the shell covers the layout without vacuum-packing every cell or inventing empty AABB corners, while **box** is the simpler bounding-shell alternative kept for comparison, and both grow an engine nozzle per rear engine cell. Two prefabs to start from: a **fighter** (excellent thrust-to-mass, deliberately tight power budget, zero cargo) and a **trader** (one gun, a shield, four cargo bays, moderate movement). Three views: **internal** (the blocks), **exterior** (the generated hull + hull color), and **performance** (blocks tinted by heat contribution, plus the live stats panel). Stats make the trade-offs concrete: mass, speed and turn (thrust-to-mass), power generation/use/balance, heat, shield HP, weapon DPS, cargo capacity, and flight range — add cargo and watch speed drop; add engines and watch power and range pay for it.

FlightRig's vertical strafe and roll (the 4th/5th DOF beyond translate+look) have two switchable input modes ("Vertical/roll input" in the GUI):
- **Sliders** (default) — two persistent, always-visible edge sliders, each paired with the same hand as its "primary" stick so both can be held at once: vertical strafe sits on the left with the move stick, roll sits on the right with the look stick (thrust + roll held simultaneously is exactly what Rotor Gates asks for). The roll slider repositions itself at runtime to clear the GUI panel's actual measured bottom edge — the panel's height varies a lot by scene and can reach nearly full-screen on the busiest courses, so a fixed position would eventually get fought over. Touch anywhere on a slider to jump the handle there and hold it deflected to sustain a rate — roll is rate control here, not position control, so holding a pose is enough; no continued motion needed.
- **Fingers** — the original multi-touch gestures: a second finger in the move-stick zone strafes up/down, a second finger on the look stick twists to roll (this one maps the twist angle directly to roll angle — precise, but physically hard to sustain past a partial turn since twisting fingers around each other runs out of comfortable range fast).

FlightRig also draws a fixed crosshair at screen center — a stable "dead ahead" reference for lining up a roll against a rotating target (Rotor Gates) or just orienting in general. It renders above the GUI panel (z-index above lil-gui's own) since the panel often covers screen center, but stays `pointer-events: none` throughout so it never affects GUI interaction.

The whole app also blocks double-tap and pinch zoom at the event level (`core/prevent-zoom.ts`) — iOS Safari ignores the viewport meta's `user-scalable=no` since iOS 10, so `touch-action`/viewport settings alone can't be trusted to stop an accidental zoom from derailing the touch controls.

Each scene is a self-contained module in `src/scenes/`; add a new one by
creating a file that exports a `TestScene` (see `src/scenes/types.ts`) and
registering it in `src/scenes/index.ts`. Scenes are lazily loaded, so a new
scene's dependencies don't bloat the initial load.

Shared infrastructure lives in `src/core/`:

- `renderer-context.ts` — renderer, camera, and touch-friendly `OrbitControls` (one-finger rotate, two-finger pinch/pan), with resize handling.
- `ticker.ts` — the render loop; pauses automatically when the tab/app is backgrounded to save battery.
- `fps-meter.ts` — a rolling FPS readout in the topbar; tap it to open a detailed Stats.js panel (FPS/MS/MB).
- `device-info.ts` — an on-demand panel (ⓘ button) showing viewport size, DPR, GPU vendor/renderer, touch points, network info, etc. — handy for reporting bugs from a real device.

## Mobile-specific details

- Viewport meta uses `viewport-fit=cover` with safe-area-aware CSS so the UI clears notches/home indicators.
- `touch-action: none` + `overscroll-behavior: none` stop pull-to-refresh and pinch-zoom from fighting the 3D controls.
- Device pixel ratio is capped at 2 to keep fill-rate sane on high-DPI phones.
- The render loop pauses on `visibilitychange` so a backgrounded tab doesn't drain the battery.
- `vite.config.ts` sets `server.host: true` so `npm run dev` is reachable from a phone on the same network (check the "Network" URL Vite prints).

## Getting started

```bash
npm install
npm run dev       # local dev server; open the printed Network URL on your phone
npm run build      # type-check + production build to dist/
npm run preview    # serve the production build locally
```

## Deploying to Cloudflare Workers

This repo is set up for [Workers Static Assets](https://developers.cloudflare.com/workers/static-assets/)
via `wrangler.jsonc` — no server-side code required, it just serves `dist/`.

```bash
npx wrangler login   # first time only
npm run deploy        # builds, then runs `wrangler deploy`
```

`wrangler.jsonc` also sets `build.command: "npm run build"`, so `wrangler
deploy` runs the Vite build itself before uploading `dist/`. That means if
you connect this repo to a Worker via Cloudflare's Git integration (Workers
Builds), you don't need to configure a Build command in the dashboard at
all — leave it as whatever it defaults to and just make sure the Deploy
command is `npx wrangler deploy`; wrangler builds fresh on every deploy.

`wrangler.jsonc` sets `name: "mobile-3d-testing"`; change it if you want a
different Workers subdomain (`<name>.<your-subdomain>.workers.dev`), or wire
up a custom domain/route in the Cloudflare dashboard afterwards.
